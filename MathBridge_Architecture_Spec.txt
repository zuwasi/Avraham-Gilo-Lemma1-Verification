================================================================================
   1. PRODUCT VISION
================================================================================

   A web application where users upload a research paper (PDF), type
   mathematical questions in plain English, and receive accurate,
   CAS-verified answers with interactive visualizations — powered by
   Ampcode + Wolfram Mathematica under the hood.

   What ChatGPT does:     Natural language in, natural language out
   What Mathematica does:  Exact symbolic/numerical computation
   What MathBridge does:   Both. Together. Automatically.

   Target users:
     - Academic researchers (like Prof. Gilo)
     - Graduate students working with proofs
     - Research teams needing to verify/extend published results
     - Anyone who needs accurate math but can't code in Wolfram Language

================================================================================
   3. USER FLOW
================================================================================

   Step 1: User uploads PDF
     - Browser sends PDF to backend
     - Backend stores PDF in session temp directory
     - Backend confirms upload to user

   Step 2: User types question in English
     Example: "Verify Lemma 1 in this paper. Then check if it applies
               to CRRA utility. Show me interactive plots."

   Step 3: Backend sends to Ampcode
     - Constructs prompt with PDF path reference
     - Calls amp CLI with stream-json mode
     - Wolfram-Mathematica skill is pre-loaded
     - CandleKeep provides Wolfram syntax reference

   Step 4: Ampcode generates Wolfram code
     - Reads PDF, extracts equations
     - Writes Wolfram Language code
     - Validates syntax (6-point check)
     - Runs self-consistency verification

   Step 5: Backend executes Wolfram code
     - Sends generated code to Wolfram Engine
     - Collects numerical results, plot data
     - Feeds results back to Ampcode for interpretation

   Step 6: Ampcode explains in plain English
     - Receives computation results
     - Writes human-readable explanation
     - Notes any errors, corrections, or insights
     - Produces final .nb file for download

   Step 7: User sees results
     - Plain English answer in chat
     - Interactive plots rendered in browser
     - Download button for .nb file
     - Can ask follow-up questions

================================================================================
   4. COMPONENTS DETAIL
================================================================================

   4.1  FRONTEND (Browser)
   ────────────────────────

     Framework:    React or plain HTML/JS (keep simple for MVP)

     Pages:
       /           Landing page with upload + chat
       /session/:id  Active session with conversation

     UI Elements:
       - PDF upload area (drag-and-drop)
       - Chat interface (messages + streaming responses)
       - Plot panel (interactive Plotly charts with sliders)
       - Download panel (.nb file, .txt summary, .pdf report)
       - Session history sidebar

     Key Features:
       - Streaming responses (show text as it arrives)
       - LaTeX rendering for equations (MathJax or KaTeX)
       - Interactive sliders for parameter exploration
       - Mobile-responsive layout

   4.2  BACKEND (Web Server)
   ──────────────────────────

     Framework:    FastAPI (Python) — preferred because wolframclient
                   is Python, so one language for everything

     Endpoints:
       POST /upload          Upload PDF, return session_id
       POST /ask             Send question, get streaming response
       GET  /download/:id    Download .nb or .txt file
       GET  /session/:id     Get conversation history
       WS   /ws/:id          WebSocket for streaming responses

     Session Management:
       - Each upload creates a session
       - PDF stored in /tmp/sessions/{session_id}/
       - Conversation history stored in memory or SQLite
       - Sessions expire after 24 hours

     Ampcode Integration:
       - Spawns amp CLI process per session
       - Uses --stream-json for real-time output
       - Uses --stream-json-input for multi-turn conversation
       - Pre-loads Wolfram-Mathematica skill via AGENTS.md
       - Points CandleKeep to shared reference library

     Wolfram Integration:
       - Uses wolframclient Python package
       - Or calls wolframscript as subprocess
       - Executes code blocks extracted from Ampcode output
       - Returns results to Ampcode for interpretation

   4.3  AMPCODE LAYER
   ───────────────────

     Already built. No new development needed.

     Components used:
       - Wolfram-Mathematica Skill (paper-to-notebook pipeline)
       - CandleKeep Library (60+ Wolfram Language reference docs)
       - 6-Point Syntax Validator
       - Phase 4B Self-Consistency Verification
       - Phase 4C Plain-English Proof Generation

     Configuration (AGENTS.md for MathBridge sessions):
       - Always load wolfram-mathematica skill
       - Always load candlekeep skill
       - Set working directory to session temp folder
       - Allow file read/write in session folder only

   4.4  WOLFRAM ENGINE (Server)
   ─────────────────────────────

     Deployment:
       - Wolfram Engine installed on same server (or dedicated box)
       - Runs as persistent kernel (avoids startup latency)
       - wolframclient connects via local socket

     Capabilities:
       - Symbolic computation (Solve, DSolve, Integrate)
       - Numerical computation (FindRoot, NIntegrate, NSolve)
       - Optimization (NMinimize, FindMinimum)
       - Visualization (Plot, Plot3D, Manipulate data export)
       - Linear algebra, statistics, probability

     Output Formats:
       - Numerical results (JSON)
       - Plot data (exported as PNG or SVG for web display)
       - .nb files (for download)
       - .wl scripts (for reproducibility)

================================================================================
   5. AMPCODE CLI API INTEGRATION
================================================================================

   The backend communicates with Ampcode using the streaming JSON API.

   Startup command:

     amp --execute --stream-json --stream-json-input \
         --prompt "You are MathBridge. User has uploaded a research \
         paper at {pdf_path}. Use the wolfram-mathematica skill. \
         Answer mathematical questions in plain English. Execute \
         all computations via Wolfram Engine. Verify all results."

   Sending user messages (stdin):

     {"type":"user","message":{"role":"user","content":[
       {"type":"text","text":"Verify Lemma 1 and check CRRA utility"}
     ]}}

   Receiving responses (stdout):

     {"type":"assistant","message":{"role":"assistant","content":[
       {"type":"text","text":"I'll analyze Lemma 1 from the paper..."}
     ]}}

   The backend parses each JSON line, extracts:
     - Text content -> send to frontend chat
     - Wolfram code blocks -> execute via Wolfram Engine
     - File creation events -> offer as downloads
     - Tool calls -> track progress for UI indicators

================================================================================
   6. INTERACTIVE PLOTS (Web Alternative to Manipulate)
================================================================================

   Problem: Mathematica's Manipulate panels require a local Mathematica
   installation. Users without Mathematica cannot use them.

   Solution: Convert Manipulate specifications to web-based interactive
   plots using Plotly.js with sliders.

   Process:
     1. Ampcode generates Manipulate specification
     2. Backend extracts parameter ranges and plot function
     3. Wolfram Engine pre-computes plot data across parameter grid
     4. Backend sends data + slider config to frontend
     5. Frontend renders interactive Plotly chart with sliders
     6. Slider changes interpolate pre-computed data (instant response)
     7. Full .nb with native Manipulate available for download

   Example output for insurance demand:
     - Slider: gamma (0.1 to 15.0)
     - Slider: H/W ratio (0.1 to 0.95)
     - Slider: q loading factor (1.01 to 3.0)
     - Plot: alpha*(P) curve with peak marker
     - Text: "At these parameters, Lemma 1(ii) applies. Demand is
              monotonically decreasing."

================================================================================
   7. SECURITY AND ISOLATION
================================================================================

   PDF Handling:
     - PDFs stored in isolated session directories
     - Deleted after 24 hours or on session close
     - Never shared between sessions
     - Never sent to external search engines (unlike ChatGPT Deep
       Research, which found Prof. Gilo's old paper online)

   Ampcode Sandboxing:
     - Each session runs its own amp process
     - File access restricted to session directory
     - No network access from Wolfram Engine (no web scraping)
     - No access to other users' data

   User Data:
     - No user accounts required for MVP (session-based)
     - No conversation data stored permanently
     - No training on user data (Ampcode Enterprise: zero retention)

   12. RISKS AND MITIGATIONS
================================================================================

   Risk: Wolfram Engine license cost too high for early stage
   Mitigation: ESL is already a Wolfram Technology Partner for Israel.
   Partner pricing applies. Engine licensing is a known quantity.

   Risk: Ampcode CLI API changes or breaks
   Mitigation: stream-json is a documented interface. Pin amp version.
   Ampcode SDK is listed in their manual — may provide more stable API.

   Risk: Complex papers produce incorrect Wolfram code
   Mitigation: The 6-point syntax validator + self-consistency check
   already handle this. Add a "confidence score" to each answer.

   Risk: Slow response time (LLM + CAS execution)
   Mitigation: Full paper analysis takes 2-5 minutes. Set user
   expectations with progress indicators. Follow-up questions are fast
   (30-60 seconds) because context is already loaded.

   Risk: Limited to math/physics papers
   Mitigation: Start with economics/finance/physics (proven). Expand
   to engineering, biology (SIR models), chemistry as skill matures.

   12.6  IMPLEMENTATION (Python / FastAPI)
   ────────────────────────────────────────

     Dependencies:
       pip install stripe

     Checkout flow:

       import stripe
       stripe.api_key = os.environ["STRIPE_SECRET_KEY"]

       @app.post("/billing/checkout")
       async def create_checkout(pack: str, user_id: str):
           prices = {
               "credits_10":  {"amount": 1000, "credits": 2},
               "credits_25":  {"amount": 2500, "credits": 6},
               "credits_50":  {"amount": 5000, "credits": 14},
               "credits_100": {"amount": 10000, "credits": 30},
           }
           session = stripe.checkout.Session.create(
               payment_method_types=["card"],
               line_items=[{
                   "price_data": {
                       "currency": "usd",
                       "unit_amount": prices[pack]["amount"],
                       "product_data": {"name": f"MathBridge {pack}"},
                   },
                   "quantity": 1,
               }],
               mode="payment",
               metadata={"user_id": user_id, "credits": prices[pack]["credits"]},
               success_url="https://mathbridge.app/success",
               cancel_url="https://mathbridge.app/pricing",
           )
           return {"checkout_url": session.url}

     Webhook handler:

       @app.post("/billing/webhook")
       async def stripe_webhook(request: Request):
           payload = await request.body()
           sig = request.headers.get("stripe-signature")
           event = stripe.Webhook.construct_event(
               payload, sig, os.environ["STRIPE_WEBHOOK_SECRET"]
           )
           if event["type"] == "checkout.session.completed":
               session = event["data"]["object"]
               user_id = session["metadata"]["user_id"]
               credits = int(session["metadata"]["credits"])
               add_credits_to_user(user_id, credits)
           return {"status": "ok"}
